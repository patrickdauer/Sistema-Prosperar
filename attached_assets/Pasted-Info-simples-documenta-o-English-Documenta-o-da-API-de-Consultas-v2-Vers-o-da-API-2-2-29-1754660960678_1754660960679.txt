Histórico de mudanças
A API está em constante evolução e novas consultas são adicionadas frequentemente, com o cuidado de introduzir apenas alterações incrementais para evitar que qualquer integração anterior deixe de funcionar.
Você é notificado por email sempre que uma mudança ocorre e pode verificar o histórico completo de mudanças no link abaixo:
https://api.infosimples.com/mudancas
Migrando da v1 para a v2
Caso você não esteja migrando para a v2, você pode pular a leitura desse item.
Caso você não queria migrar para a v2, você não precisa modificar a sua integração. Manteremos a v1 funcionando.
A versão 2 da API da Infosimples introduz melhorias que facilitam a integração com outras aplicações. Entre outras coisas, a forma como os resultados e arquivos de visualização de consulta são organizados foi melhorada e, por isso, a interface da v2 é incompatível com a v1.
Abaixo estão indicados os pontos sensíveis que você deve verificar para migrar da v1 para a v2:
•	A rota de requisição mudou de https://api.infosimples.com/api/v1/ para https://api.infosimples.com/api/v2/consultas/.
•	O campo header do JSON de retorno sofreu alterações. 
o	As seguintes chaves foram removidas: client, token, credits, price_extra, has_limit, limit, used, limit_price, used_price, cache_hit e cached_at.
o	Algumas chaves foram adicionadas. Veja aqui o formato completo do campo header na v2. 
•	O campo data do JSON de retorno passou a ser sempre um array. Ou seja, mesmo quando a consulta retornar um único resultado, esse campo será um array, de modo que agora a interface de retorno é mais consistente.
•	O código de resposta 201 não é mais utilizado. Na v2, consultas que retornarem um ou mais resultados utilizarão sempre o código 200. O campo data_count continua disponível para verificação da quantidade de resultados retornados.
•	A mensagem de resposta associada ao código 200 passou a ser "A requisição foi processada com sucesso".
•	O campo errors do JSON de retorno passou a ser sempre um array (não pode mais ser null).
•	Para facilitar a associação entre os dados parseados e os respectivos arquivos de visualização de consulta, cada elemento dentro do campo data agora possui a sua própria chave site_receipt com a URL direta do arquivo de visualização (pode ser null).
•	O campo receipt não existe mais. Agora, todas as URLs de arquivos de visualização de consulta podem ser obtidas, de forma agrupada, através do novo campo site_receipts, que é sempre um array.
Formato de requisição
Método HTTP	POST
URL da consulta	https://api.infosimples.com/api/v2/consultas/servico 
Formato do retorno	JSON
Substitua servico pelo serviço que será consultado, por exemplo receita-federal/cnpj. Você encontra o servico na documentação específica da respectiva API.
Parâmetros de configuração
Esses parâmetros são comuns a todas APIs. Uma API pode solicitar outros parâmetros que estarão especificados em sua documentação.
* Parâmetro obrigatório
Parâmetro	Descrição
token *	Chave de acesso da API. É com o token que a API reconhece e autoriza quem está fazendo a consulta.
timeout	Duração máxima aceitável para realizar a consulta, em segundos. Valor mínimo: 15. Valor máximo: 600. 
Importante: é necessário configurar adequadamente o timeout da sua biblioteca HTTP para que a requisição não seja interrompida antes do timeout solicitado para a API da Infosimples. 
ignore_site_receipt	Quando informado com o valor 1, o campo site_receipts é retornado vazio. Por padrão, site_receipts são sempre retornados.. 
Estrutura de resposta
Uma resposta da API possui os seguintes campos:
Campo	Tipo	Descrição
code	number	Código de resposta da API.
code_message	string	Mensagem correspondente ao código de resposta da API.
data	array	Dados estruturados a partir do resultado da automação da consulta na fonte original. É sempre um array.
data_count	number	Quantidade de resultados retornados, ou seja, a quantidade de elementos no campo data.
errors	array	Lista de mensagens (String) que podem explicar com mais detalhes a causa de algum erro. É sempre um array.
site_receipts	array	Lista de URLs dos documentos HTMLs ou PDFs emitidos pela fonte original ou gerados pela Infosimples (em alguns casos a fonte não gera um arquivo adequado para visualização). 
Os arquivos ficam disponíveis por 7 dias corridos a partir da data da consulta. É recomendado que você crie rotinas para baixar os arquivos e armazená-los em sua própria infraestrutura. 
Caso você não tenha interesse em obter esses arquivos, você pode simplesmente adicionar o parâmetro ignore_site_receipt=1 na sua requisição. 
header	object	Cabeçalho com os seguintes campos:
•	api_version: versão da API utilizada na requisição.
•	product: nome do produto da Infosimples.
•	service: nome do serviço solicitado.
•	parameters: parâmetros específicos do serviço enviados na requisição.
•	client_name: nome do cliente.
•	token_name: nome do token usado na requisição.
•	billable: indica se houve cobrança pela requisição.
•	price: preço final do processamento de requisição.
•	requested_at: data e hora da requisição.
•	elapsed_time_in_milliseconds: duração total da requisição, em milissegundos.
•	remote_ip: endereço IP a partir do qual a requisição foi originada.
•	signature: valor que pode ser utilizado pela Infosimples para garantir a integridade e a autenticidade do retorno da API.

Boas práticas
Aqui são apresentadas sugestões técnicas simples que podem evitar diversos problemas comuns.
Retentativas de consulta
O retorno de uma consulta contém um código de resposta cuja função é identificar o tipo de resultado recebido, de modo que todos os códigos possíveis devem ser tratados.
Em alguns casos (mas não em todos!), pode ser necessário repetir uma ou mais vezes a mesma consulta, visando a obtenção do resultado desejado. Por exemplo, uma consulta com retorno de código 605 (A consulta não foi realizada dentro do tempo de limite de timeout especificado.) não é cobrada e pode ser efetuada novamente, uma vez que tipicamente o erro é transitório e há possibilidade de sucesso em uma nova tentativa. Por outro lado, uma consulta cujo código de retorno é 620 (O site ou aplicativo de origem emitiu um erro que provavelmente não mudará em breve para esta consulta. Leia-o para saber mais.) é cobrada e, em geral, não deve ser efetuada novamente em um curto intervalo de tempo.
Por fim, estabeleça um limite máximo de tentativas para uma consulta e não permita que uma requisição seja repetida indefinidamente em loop, uma vez que isso pode levar a faturas com valores altamente fora do planejado desnecessariamente.
CORS (Cross-Origin Resource Sharing)
Você não deve integrar a API da Infosimples diretamente no frontend da sua aplicação.
Essa abordagem possivelmente gerará um "erro de CORS", pois os servidores da API da Infosimples não podem ser acessados em um browser através de um site que não seja o da própria Infosimples. Esse é um comportamento programado intencionalmente e visa promover segurança nas integrações de clientes. Você precisa construir um web service que é acionado pela sua aplicação e que, por sua vez, aciona a API da Infosimples. Desta forma, a integração será mais segura e o seu token da Infosimples não ficará exposto em sua aplicação.
Timeout de cliente HTTP
A API da Infosimples aceita o parâmetro opcional timeout, cuja função é indicar o quanto você está disposto a aguardar pelo processamento da automação, para que a API da Infosimples possa então responder com código 605 caso esse tempo seja excedido. No entanto, esse parâmetro não deve ser confundido com o timeout de rede (open timeout e read timeout) do cliente HTTP da linguagem de programação que você estiver usando, o qual indica o quanto o seu próprio cliente HTTP irá aguardar por uma resposta do servidor.
Caso o timeout do cliente HTTP seja inferior ao timeout enviado na requisição, o seu programa eventualmente lançará algum tipo de exceção.
Integrações avançadas
API assíncrona
A API assíncrona permite que você faça uma requisição sem aguardar pelo seu retorno com uma conexão HTTP ativa.
Recomendamos o uso da API assíncrona para processamentos de lotes ou de monitoramentos. Saiba mais no link abaixo:
https://api.infosimples.com/consultas/docs/assincrona
Criptografia
Os parâmetros enviados para a API da Infosimples podem ser encriptados. Veja como no link abaixo:
https://api.infosimples.com/consultas/docs/criptografia
Certificados digitais
Algumas APIs disponibilizadas pela Infosimples aceitam certificados digitais PKCS12 (conhecidos popularmente como certificados A1) como parâmetros de entrada. Se você precisa integrar com uma API desse tipo, visite o link abaixo:
https://api.infosimples.com/consultas/docs/certificados
Informações da conta
Use este endpoint para consultar informações da conta, tais como saldo e consumo. Saiba mais no link abaixo:
https://api.infosimples.com/consultas/docs/conta
Informações de um serviço
Disponibilizamos um endpoint que retorna status e outras informações de um serviço. Saiba mais no link abaixo:
https://api.infosimples.com/consultas/docs/api-status
OpenAPI, Swagger e Postman
Disponibilizamos um arquivo no formato OpenAPI com as informações de todas as consultas da Infosimples, que pode ser usado no Postman, além de sua interface web do Swagger. Saiba mais no link abaixo:
https://api.infosimples.com/consultas/docs/openapi
